\Abstract{

We present a domain-specific conversational AI assistant for PC component selection, focused on CPU and GPU recommendations. General-purpose LLMs often hallucinate component specifications — our system grounds every response in passages retrieved from curated hardware reviews via a Retrieval-Augmented Generation (RAG) pipeline. The system features LLM-driven query rewriting for multi-turn coherence, semantic routing between separate CPU and GPU indexes, and citation-aware generation with Mistral-7B-Instruct \cite{jiang2023mistral}. Qualitative evaluation shows strong performance on specific product comparisons while revealing limitations on queries about components absent from the review corpus.

}

\section*{Introduction}

Choosing PC components is a common technical challenge for newcomers, yet general-purpose LLMs often produce incorrect information — hallucinated socket types, non-existent models, or stale benchmark figures. We built a conversational assistant grounded in real hardware review content that helps users select CPUs and GPUs based on their needs and budget. The system uses Retrieval-Augmented Generation to cite sources for every claim and maintains multi-turn conversation context so users can refine their requirements across several exchanges.

%------------------------------------------------

\section*{Dataset}

We assembled a corpus of 49 hardware review articles: 29 covering CPUs and 20 covering GPUs, sourced from TechPowerUp, TechSpot, Tom's Hardware, Eurogamer, PC Gamer, PCMag, and Digital Trends. Articles were scraped and converted to Markdown using a custom scraper. Each article was then split into chunks at H2-level section boundaries (e.g. \textit{Gaming Performance}, \textit{Thermals}, \textit{Conclusion}), with chunks shorter than 80 characters discarded as uninformative. This produced two separate corpora — one per component category — used to build independent retrieval indexes. Table~\ref{tab:dataset} summarises the corpus.

\section*{System}

\subsection*{Overview}
The pipeline (Figure~\ref{fig:pipeline}) consists of four stages: query routing, query rewriting, retrieval, and generation. All stages share two frozen models: \texttt{BAAI/bge-base-en-v1.5} (768-dim) for dense retrieval and \texttt{mistralai/Mistral-7B-Instruct-v0.3} for rewriting and generation.

\subsection*{Indexing}
Each corpus was embedded with \texttt{BAAI/bge-base-en-v1.5} (batch size 32, L2-normalised) and stored in a FAISS \texttt{IndexFlatIP} index. Because embeddings are normalised, inner product equals cosine similarity, making retrieval scores directly interpretable as relevance confidence.

\subsection*{Query Routing}
A lightweight routing step selects which domain index to query. Both the incoming query and a fixed keyword description for each domain (\textit{"processor CPU cores threads clock speed gaming workloads"} for CPU; \textit{"graphics card GPU VRAM ray tracing frames rendering"} for GPU) are embedded with \texttt{all-MiniLM-L6-v2} (384-dim). The cosine similarity between query and each keyword centroid determines which index is queried. Any index scoring above 0.4 is included; if none pass, the highest-scoring index is used as a fallback.

\subsection*{Query Rewriting}
In multi-turn conversations, pronouns and ellipsis make queries ambiguous for retrieval (e.g.\ \textit{"how does it compare to the other one?"}). Before retrieval, Mistral-7B rewrites the user question into a standalone search query conditioned on the conversation history. Rewriting uses greedy decoding (temperature = 0, max 100 new tokens) to ensure deterministic, compact output.

\subsection*{Retrieval and Generation}
The rewritten query is embedded and the top $k=3$ chunks above a cosine similarity threshold of 0.5 are retrieved from each selected index. Retrieved chunks are injected into the prompt with inline citation instructions so the model attributes claims to numbered sources. Mistral-7B-Instruct-v0.3 generates the final response with temperature 0.7 and a maximum of 512 new tokens running in fp16 on CUDA. Up to 6 conversation turns (12 messages) are retained in context.

\section*{Related work}
RAG-based systems demonstrate that injecting retrieved passages substantially improves factual accuracy on knowledge-intensive tasks, yet performance drops when key facts are absent from the corpus \cite{lewis2020rag,izacard2021leveraging}. Instruction-tuned models offer conversational control but inherit the same grounding limitations without authoritative retrieval \cite{ouyang2022training}. Query rewriting has been shown to significantly improve multi-turn retrieval quality by resolving coreferences before the search step \cite{ma2023queryrewrite}. Tools like PCPartPicker \cite{pcpartpicker2026compatibility} demonstrate reliable rule-based compatibility checking but lack conversational flexibility; our system trades deterministic rule coverage for broader, review-grounded recommendation quality.



\section*{Results}

We evaluated the assistant qualitatively across three query types: specific product comparisons, budget recommendation requests, and out-of-scope or out-of-corpus questions.

\textbf{Works well.} When a reviewed product is queried directly (e.g. \textit{"Is the Ryzen 7 9800X3D worth it for gaming?"}), the system reliably retrieves the relevant review sections and generates accurate, cited answers. Multi-turn follow-ups (e.g. \textit{"How does it compare thermally?"}) resolve correctly via query rewriting — without it, naive retrieval on the anaphoric follow-up returns irrelevant chunks.

\textbf{Degrades gracefully.} For budget queries (e.g. \textit{"Best GPU under \$300"}), the system retrieves buyer-guide sections with ranked recommendations and scores them against the budget. The answers are useful but less precise, as prices in reviews can be outdated.

\textbf{Fails.} The system cannot answer questions about products absent from the corpus (e.g. unreleased or niche GPUs) — it correctly refuses rather than hallucinating, but this is a hard coverage limit. Routing also occasionally misfires: queries phrased as general computing questions (e.g. \textit{"Will 16GB RAM bottleneck performance?"}) are sometimes routed to both indexes, inflating the context and diluting relevance.

Formal automatic evaluation (e.g. Recall@k on a held-out query set) is left for future work given the absence of a labelled QA benchmark for this narrow domain.
